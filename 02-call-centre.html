Call Centre

Metrics in this section are calculated from call centre activity and topic data.

Primary tables

  • call_centre.weekly_activity_report
  • call_centre.weekly_topic_dump

Notes

Call centre metrics are useful for understanding support demand, service responsiveness, caller wait times, average call duration, and common reasons for contact.

The current cookbook primarily uses weekly call centre data, so call centre metrics are often reported at a weekly level or aggregated across weeks.

Calls Accepted

Description

The number of calls accepted by the call centre during a reporting period.

In the notebook, this is visualized by week using calls_accepted from the weekly activity report.

This metric is useful for tracking call centre demand and understanding how contact volume changes over time.

Plain-language alternate names

  • Calls received
  • Accepted calls
  • Weekly call volume
  • Call centre demand
  • Number of calls handled by the queue

Data source

  • Schema: call_centre
  • Table: weekly_activity_report
  • Key fields:
    • date_range_start
  • date_range_end
  • calls_accepted

Calculation logic

Use calls_accepted by reporting period, or sum calls_accepted across reporting periods for a total.

Example SQL by week

SELECT
date_range_start,
date_range_end,
CAST(calls_accepted AS BIGINT) AS calls_accepted
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Example SQL for total calls accepted

SELECT
SUM(CAST(calls_accepted AS BIGINT)) AS total_calls_accepted
FROM call_centre.weekly_activity_report
WHERE date_range_start BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

Interpretation notes

Calls accepted is a volume metric. It may not be identical to calls answered, depending on call centre definitions and routing logic.

Calls Answered

Description

The number of calls answered by call centre agents during a reporting period.

This metric is used as a weighting variable when calculating weighted average call length and weighted average delay.

Calls answered may differ from calls accepted depending on call centre definitions, routing, abandonment, or reporting logic.

Plain-language alternate names

  • Answered calls
  • Calls handled
  • Agent-answered calls
  • Completed calls
  • Calls taken by agents

Data source

  • Schema: call_centre
  • Table: weekly_activity_report
  • Key fields:
    • date_range_start
  • date_range_end
  • calls_answered

Calculation logic

Use calls_answered by reporting period, or sum calls_answered across reporting periods for a total.

Example SQL by week

SELECT
date_range_start,
date_range_end,
CAST(calls_answered AS BIGINT) AS calls_answered
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Example SQL for total calls answered

SELECT
SUM(CAST(calls_answered AS BIGINT)) AS total_calls_answered
FROM call_centre.weekly_activity_report
WHERE date_range_start BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

Interpretation notes

Calls answered is commonly used as the weighting field when combining weekly average call length or delay values.

Average Call Length

Description

The average length of answered calls.

In the notebook, the overall average call length is calculated as a weighted mean of avg_call_length_seconds, weighted by calls_answered.

This is preferable to a simple average of weekly averages because weeks with more answered calls should contribute more to the overall result.

Plain-language alternate names

  • Average call duration
  • Average handle time
  • Average length of call
  • Typical call length
  • Average time on call

Data source

  • Schema: call_centre
  • Table: weekly_activity_report
  • Key fields:
    • avg_call_length_seconds
  • calls_answered

Calculation logic

Calculate a weighted average of weekly average call length, weighted by the number of answered calls.

Example SQL

SELECT
SUM(
  CAST(avg_call_length_seconds AS DOUBLE)
  * CAST(calls_answered AS DOUBLE)
)
/ NULLIF(SUM(CAST(calls_answered AS DOUBLE)), 0) AS weighted_avg_call_length_seconds
FROM call_centre.weekly_activity_report
WHERE date_range_start BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

Example SQL by week

SELECT
date_range_start,
date_range_end,
CAST(avg_call_length_seconds AS DOUBLE) AS avg_call_length_seconds,
CAST(calls_answered AS BIGINT) AS calls_answered
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Interpretation notes

Use a weighted average rather than a simple average of weekly averages.

A simple average of weekly averages may give too much influence to low-volume weeks.

Average Delay

Description

The average delay before calls are answered.

In the notebook, this is calculated as a weighted mean of avg_delay_seconds, weighted by calls_answered.

This metric is useful for understanding caller wait times and call centre responsiveness.

Plain-language alternate names

  • Average wait time
  • Average delay before answer
  • Caller wait time
  • Time to answer
  • Average queue delay

Data source

  • Schema: call_centre
  • Table: weekly_activity_report
  • Key fields:
    • avg_delay_seconds
  • calls_answered

Calculation logic

Calculate a weighted average of weekly average delay, weighted by the number of answered calls.

Example SQL

SELECT
SUM(
  CAST(avg_delay_seconds AS DOUBLE)
  * CAST(calls_answered AS DOUBLE)
)
/ NULLIF(SUM(CAST(calls_answered AS DOUBLE)), 0) AS weighted_avg_delay_seconds
FROM call_centre.weekly_activity_report
WHERE date_range_start BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

Example SQL by week

SELECT
date_range_start,
date_range_end,
CAST(avg_delay_seconds AS DOUBLE) AS avg_delay_seconds,
CAST(calls_answered AS BIGINT) AS calls_answered
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Interpretation notes

Use a weighted average rather than a simple average of weekly averages.

Average delay should be interpreted alongside call volume. A low-volume week can have unusual wait times that should not dominate the overall result.

Weekly Call Centre Activity

Description

A weekly summary of call centre activity.

The notebook currently uses this to visualize calls_accepted by week, but the same table can also support reporting on answered calls, average call length, and average delay.

This metric family is useful for operational monitoring and trend reporting.

Plain-language alternate names

  • Weekly call centre summary
  • Weekly call activity
  • Call centre weekly report
  • Weekly service volume
  • Weekly caller demand

Data source

  • Schema: call_centre
  • Table: weekly_activity_report
  • Key fields:
    • date_range_start
  • date_range_end
  • calls_accepted
  • calls_answered
  • avg_call_length_seconds
  • avg_delay_seconds

Calculation logic

Select weekly reporting fields from weekly_activity_report.

Example SQL

SELECT
date_range_start,
date_range_end,
CAST(calls_accepted AS BIGINT) AS calls_accepted,
CAST(calls_answered AS BIGINT) AS calls_answered,
CAST(avg_call_length_seconds AS DOUBLE) AS avg_call_length_seconds,
CAST(avg_delay_seconds AS DOUBLE) AS avg_delay_seconds
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Interpretation notes

Weekly call centre activity is useful as a base table for dashboards, but higher-level metrics should be clearly defined to avoid mixing counts and averages.

Call Topic Count

Description

The number of call records associated with each call topic.

In the notebook, the field statobject is cleaned using whitespace squishing and then counted. The top 10 topics are displayed.

This metric helps answer: “What are people calling about?”

Plain-language alternate names

  • Call reasons
  • Call topics
  • Reasons for calling
  • Top caller issues
  • Contact drivers

Data source

  • Schema: call_centre
  • Table: weekly_topic_dump
  • Key fields:
    • statobject

Calculation logic

Clean the statobject field by trimming leading and trailing whitespace and replacing repeated whitespace with a single space. Then count records by cleaned topic.

Example SQL

SELECT
REGEXP_REPLACE(TRIM(statobject), '\\s+', ' ') AS call_topic,
COUNT(*) AS topic_count
FROM call_centre.weekly_topic_dump
GROUP BY REGEXP_REPLACE(TRIM(statobject), '\\s+', ' ')
ORDER BY topic_count DESC;

Interpretation notes

Call topic counts depend on how topics are coded in the source system. Topic naming, duplicates, and inconsistent spelling can affect results.

Call Topic Share

Description

The percentage of all topic records represented by each call topic.

This is calculated by counting each topic and dividing by the total number of topic records.

It is useful for understanding the relative importance of each call driver.

Plain-language alternate names

  • Share of calls by topic
  • Percentage of calls by reason
  • Topic distribution
  • Call reason mix
  • Contact driver share

Data source

  • Schema: call_centre
  • Table: weekly_topic_dump
  • Key fields:
    • statobject

Calculation logic

Count records by cleaned topic, then divide each topic count by the total count across all topics.

Example SQL

WITH topic_counts AS (
  SELECT
  REGEXP_REPLACE(TRIM(statobject), '\\s+', ' ') AS call_topic,
  COUNT(*) AS topic_count
  FROM call_centre.weekly_topic_dump
  GROUP BY REGEXP_REPLACE(TRIM(statobject), '\\s+', ' ')
)

SELECT
call_topic,
topic_count,
CAST(topic_count AS DOUBLE)
/ NULLIF(SUM(topic_count) OVER (), 0) AS topic_share
FROM topic_counts
ORDER BY topic_count DESC;

Interpretation notes

Topic share is useful for comparing relative demand across contact reasons. It should be interpreted alongside total call volume.

Top 10 Call Topics

Description

The ten most common call topics in the call centre topic data.

This is the metric currently shown in the notebook as a table of the top 10 values of statobject.

It is useful for quickly identifying the most common reasons people contact the call centre.

Plain-language alternate names

  • Top 10 call reasons
  • Most common call topics
  • Top contact drivers
  • Main reasons for calling
  • Top caller issues

Data source

  • Schema: call_centre
  • Table: weekly_topic_dump
  • Key fields:
    • statobject

Calculation logic

Count records by cleaned topic, calculate topic share, sort in descending order by topic count, and return the top 10 topics.

Example SQL

WITH topic_counts AS (
  SELECT
  REGEXP_REPLACE(TRIM(statobject), '\\s+', ' ') AS call_topic,
  COUNT(*) AS topic_count
  FROM call_centre.weekly_topic_dump
  GROUP BY REGEXP_REPLACE(TRIM(statobject), '\\s+', ' ')
),

topic_shares AS (
  SELECT
  call_topic,
  topic_count,
  CAST(topic_count AS DOUBLE)
  / NULLIF(SUM(topic_count) OVER (), 0) AS topic_share
  FROM topic_counts
)

SELECT
call_topic,
topic_count,
topic_share
FROM topic_shares
ORDER BY topic_count DESC
LIMIT 10;

Interpretation notes

Top 10 call topics are useful for quick reporting but may hide important lower-volume issues. For diagnostic work, review the full topic distribution.
