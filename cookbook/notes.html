Additional Notes

Sign-ins are not users

A sign-in count measures events. One person can sign in multiple times.

Use sign-ins when the question is about volume or system usage.

Use unique users when the question is about reach.

Daily unique users are not the same as period unique users

Summing daily unique users gives a measure of user-days, not distinct people over the full period.

For distinct people over a longer period, use a cumulative unique user field such as mtd_unique_users or ytd_unique_users.

Metrics can be reported over different timespans

Most metrics are meaningful over more than one window. The three standard reporting windows are month-to-date, year-to-date, and since the April 22, 2026 public launch. How you select a window depends on the kind of metric.

Additive metrics (sign-ins, MFA attempts, success rates, averages) are computed by aggregating daily rows, so the window is just the date bounds in the WHERE from_date BETWEEN DATE '<start>' AND DATE '<end>' clause. The same query covers any window; only the start date changes:

  • Month-to-date: first of the current month (for example, 2026-06-01)
  • Year-to-date: 2026-01-01
  • Since launch: 2026-04-22

Set the end date to the reporting date.

Cumulative unique-user metrics cannot be derived by filtering a date range, because summing daily unique_users counts user-days, not distinct people. Instead, pick the pre-computed column that matches the window and take its latest value:

  • Month-to-date: mtd_unique_users
  • Year-to-date: ytd_unique_users

Never sum these cumulative columns across rows.

There is no cumulative since-launch column. The since_apr22_unique_users field that previously provided it has been removed: the cumulative count is computed by the IBM Verify API, which sees only the most recent 366.5 days of source data, so a since-launch total silently truncates once launch is more than a year in the past. The mtd_ and ytd_ windows are unaffected because they fall inside that horizon.

This limit does not affect the additive metrics above. Those are aggregated from the daily rows we store on our side, which are retained beyond a year, so a date-bounded since-launch window over an additive metric stays correct however far back it reaches. See the data retention note in the catalog.

Weighted averages should be used for call centre averages

When calculating average call length or average delay across multiple weeks, use a weighted average based on calls_answered.

A simple average of weekly averages may give too much influence to low-volume weeks.

Internal applications should usually be excluded from external adoption reporting

The following applications are currently treated as internal:

  • Flow Application
  • GCS Migration Solution
  • GC Sign In - Profile Management App

Whether to exclude them depends on the reporting question.
