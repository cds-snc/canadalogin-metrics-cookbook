Jira (PSOM)

Metrics in this section are calculated from PSOM Jira Service Management ticket data.

The relevant Jira table is:

  • jira.psom - weekly snapshot of all PSOM tickets

Data cadence. The jira.psom table is updated once per week (every Monday at ~06:00 ET). Each run writes a full snapshot of all current tickets to a new date= partition; previous partitions are preserved as an audit trail. Always filter to date = (SELECT MAX(date) FROM jira.psom) when querying the latest state.

PSOM Metrics

Tickets by Status

The number of open PSOM Jira tickets grouped by their current workflow status, based on the latest weekly snapshot.

Alternative names:

  • ticket status breakdown
  • open tickets by status
  • PSOM ticket status distribution

Calculation. Filter to the most recent date partition and count tickets by status.

  • SQL
  • R
SELECT
    status,
    COUNT(*) AS ticket_count
FROM jira.psom
WHERE date = (SELECT MAX(date) FROM jira.psom)
GROUP BY status
ORDER BY ticket_count DESC;
psom <- tbl(con, in_schema("jira", "psom"))
latest_date <- psom |> summarise(d = max(date, na.rm = TRUE)) |> pull(d)

psom |>
  filter(as.Date(date) == as.Date(latest_date)) |>
  count(status, name = "ticket_count") |>
  arrange(desc(ticket_count))

Weekly Ticket Volume

The total number of PSOM tickets captured in each weekly snapshot, showing how the backlog size changes over time.

Alternative names:

  • total tickets per week
  • weekly PSOM ticket count
  • ticket volume trend

Calculation. Count the number of rows per date partition. Each partition is one Monday snapshot, so the result is one total per week.

  • SQL
  • R
SELECT
    date,
    COUNT(*) AS ticket_count
FROM jira.psom
GROUP BY date
ORDER BY date DESC;
tbl(con, in_schema("jira", "psom")) |>
  count(date, name = "ticket_count") |>
  arrange(desc(date))

SLA First-Response Breach Rate

The proportion of PSOM tickets that missed the “Time to first response” SLA (a 4-hour response target), based on the latest weekly snapshot.

This metric is useful for monitoring whether the team is meeting initial response commitments. Only tickets where the SLA clock was active are included; tickets with no SLA (e.g., non-service-request issue types or tickets created before the SLA scheme was configured) are excluded.

Alternative names:

  • first response SLA breach rate
  • time to first response SLA compliance
  • first-response SLA failure rate

Calculation. Filter to the most recent date partition and to tickets where sla_first_response_breached is not empty (i.e., the SLA was active). Divide the count of tickets where sla_first_response_breached = 'True' by the total count of SLA-active tickets.

  • SQL
  • R
SELECT
    COUNT(*) AS total_sla_tickets,
    SUM(CASE WHEN sla_first_response_breached = 'True' THEN 1 ELSE 0 END) AS breached_count,
    CAST(SUM(CASE WHEN sla_first_response_breached = 'True' THEN 1 ELSE 0 END) AS DOUBLE)
        / NULLIF(COUNT(*), 0) AS breach_rate
FROM jira.psom
WHERE date = (SELECT MAX(date) FROM jira.psom)
  AND sla_first_response_breached != '';
psom <- tbl(con, in_schema("jira", "psom"))
latest_date <- psom |> summarise(d = max(date, na.rm = TRUE)) |> pull(d)

psom |>
  filter(as.Date(date) == as.Date(latest_date), sla_first_response_breached != "") |>
  summarise(
    total_sla_tickets = n(),
    breached_count    = sum(as.integer(sla_first_response_breached == "True"), na.rm = TRUE),
    breach_rate       = sum(as.integer(sla_first_response_breached == "True"), na.rm = TRUE) /
                          as.double(n())
  )

Interpretation notes. The rate is a proportion between 0 and 1. Tickets with no active SLA (empty sla_first_response_breached) are excluded from both numerator and denominator. Check total_sla_tickets to understand the sample size underlying the rate.

A ticket that is still open and within its 4-hour window is recorded as not breached ("False"), the same as a ticket that met the SLA. The rate on a live snapshot is therefore a lower bound: some of those open tickets may still breach before they are answered. It is most meaningful once the snapshot’s tickets are largely resolved.

The proportion of PSOM tickets that missed the “Time to done” SLA (a 16-hour resolution target), based on the latest weekly snapshot.

This metric reflects how well the team is resolving tickets within the agreed timeframe. Only tickets where the SLA clock was active are included; tickets with no SLA configuration are excluded.

Alternative names:

  • time to done SLA breach rate
  • resolution SLA compliance
  • time-to-done SLA failure rate
