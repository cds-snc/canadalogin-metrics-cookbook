Call Centre

Metrics in this section are calculated from call centre activity and topic data.

Call centre metrics are useful for understanding support demand, service responsiveness, caller wait times, average call duration, and common reasons for contact. The source data is weekly, so these metrics are typically reported at a weekly level or aggregated across weeks.

The relevant call centre tables are:

  • call_centre.weekly_activity_report - weekly call volume and service-level summary
  • call_centre.weekly_topic_dump - individual call records with topic classifications

date_range_start and date_range_end are stored as text, so date-range filters cast them to DATE before comparing to date literals.

Activity Metrics

Metrics in this section describe call centre operations, sourced from call_centre.weekly_activity_report.

Calls Accepted

The number of calls accepted by the call centre during a reporting period.

This metric is useful for tracking call centre demand and understanding how contact volume changes over time.

Alternative names:

  • calls received
  • accepted calls
  • weekly call volume
  • call centre demand
  • number of calls handled by the queue

Calculation. Use calls_accepted by reporting period, or sum across periods for a total.

  • SQL
  • R

By week:

SELECT
    date_range_start,
    date_range_end,
    CAST(calls_accepted AS BIGINT) AS calls_accepted
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Total over a period:

SELECT
    SUM(CAST(calls_accepted AS BIGINT)) AS total_calls_accepted
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

By week:

tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  arrange(date_range_start) |>
  select(
    date_range_start,
    date_range_end,
    calls_accepted
  )

Total over a period:

tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")
  ) |>
  summarise(
    total_calls_accepted = sum(calls_accepted, na.rm = TRUE)
  )

Interpretation notes. Calls accepted may not be identical to calls answered, depending on call centre definitions and routing logic.

Average Call Length

The average length of answered calls.

The overall average call length is calculated as a weighted mean of avg_call_length_seconds, weighted by calls_answered. This is preferable to a simple average of weekly averages because weeks with more answered calls should contribute more to the overall result.

Alternative names:

  • average call duration
  • average handle time
  • average length of call
  • typical call length
  • average time on call

Calculation. Calculate a weighted average of weekly average call length, weighted by the number of answered calls.

  • SQL
  • R

Weighted average over a period:

SELECT
    SUM(
        CAST(avg_call_length_seconds AS DOUBLE)
        * CAST(calls_answered AS DOUBLE)
    )
    / NULLIF(SUM(CAST(calls_answered AS DOUBLE)), 0) AS weighted_avg_call_length_seconds
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

By week:

SELECT
    date_range_start,
    date_range_end,
    CAST(avg_call_length_seconds AS DOUBLE) AS avg_call_length_seconds,
    CAST(calls_answered AS BIGINT) AS calls_answered
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Weighted average over a period:

tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")
  ) |>
  summarise(
    weighted_avg_call_length_seconds =
      sum(avg_call_length_seconds * calls_answered, na.rm = TRUE) /
        sum(calls_answered, na.rm = TRUE)
  )

By week:

tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  arrange(date_range_start) |>
  select(
    date_range_start,
    date_range_end,
    avg_call_length_seconds,
    calls_answered
  )

Interpretation notes. Use a weighted average rather than a simple average of weekly averages. A simple average may give too much influence to low-volume weeks.

Average Delay

The average delay before calls are answered.

This metric is useful for understanding caller wait times and call centre responsiveness. Like average call length, it should be calculated as a weighted mean to avoid low-volume weeks distorting the result.

Alternative names:

  • average wait time
  • average delay before answer
  • caller wait time
  • time to answer
  • average queue delay

Calculation. Calculate a weighted average of weekly average delay, weighted by the number of answered calls.

  • SQL
  • R

Weighted average over a period:

SELECT
    SUM(
        CAST(avg_delay_seconds AS DOUBLE)
        * CAST(calls_answered AS DOUBLE)
    )
    / NULLIF(SUM(CAST(calls_answered AS DOUBLE)), 0) AS weighted_avg_delay_seconds
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';

By week:

SELECT
    date_range_start,
    date_range_end,
    CAST(avg_delay_seconds AS DOUBLE) AS avg_delay_seconds,
    CAST(calls_answered AS BIGINT) AS calls_answered
FROM call_centre.weekly_activity_report
ORDER BY date_range_start;

Weighted average over a period:

tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")

  summarise(
    weighted_avg_delay_seconds =

Abandonment Rate

The proportion of accepted calls that were abandoned before being answered.

This metric is useful for understanding how often callers give up while waiting. It is closely tied to wait times: rising delay often drives rising abandonment.

Alternative names:

  • abandon rate
  • call abandonment rate
  • drop rate
  • proportion of calls abandoned

Calculation. Divide the sum of calls_abandoned by the sum of calls_accepted over the selected reporting period. Summing the raw counts gives a volume-weighted rate; do not average the weekly calls_abandoned_pct column.

  • SQL
  • R
SELECT
    SUM(CAST(calls_abandoned AS DOUBLE))
        / NULLIF(SUM(CAST(calls_accepted AS DOUBLE)), 0) AS abandonment_rate
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';
tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")
  ) |>
  summarise(
    abandonment_rate =
      sum(calls_abandoned, na.rm = TRUE) /
        sum(calls_accepted, na.rm = TRUE)
  )

Interpretation notes. The rate is a proportion between 0 and 1. Read it together with average delay; a spike in abandonment usually coincides with longer waits.

Answer Rate

The proportion of accepted calls that were answered by an agent.

This metric is the complement of abandonment and any other non-answered outcomes. It summarizes how much of incoming demand the call centre actually served.

Alternative names:

  • answer ratio
  • proportion of calls answered
  • pickup rate
  • calls answered rate

Calculation. Divide the sum of calls_answered by the sum of calls_accepted over the selected reporting period.

  • SQL
  • R
SELECT
    SUM(CAST(calls_answered AS DOUBLE))
        / NULLIF(SUM(CAST(calls_accepted AS DOUBLE)), 0) AS answer_rate
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';
tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")
  ) |>
  summarise(
    answer_rate =
      sum(calls_answered, na.rm = TRUE) /
        sum(calls_accepted, na.rm = TRUE)
  )

Interpretation notes. The rate is a proportion between 0 and 1. A high answer rate with rising delay can still indicate strain: calls are answered eventually, but callers wait longer to reach an agent.

Service Level (Answered Within 60s)

The proportion of calls answered within 60 seconds.

This metric is a standard call centre service-level measure of how quickly the queue is cleared. The source data stores only the weekly proportion (pct_answered_within_60s), not the underlying within-60s count, so combine weeks with a weighted average rather than re-deriving from counts.

Alternative names:

  • service level
  • SLA
  • grade of service
  • percent answered within 60 seconds
  • speed of answer

Calculation. Calculate a weighted average of pct_answered_within_60s, weighted by calls_answered.

  • SQL
  • R
SELECT
    SUM(
        CAST(pct_answered_within_60s AS DOUBLE)
        * CAST(calls_answered AS DOUBLE)
    )
    / NULLIF(SUM(CAST(calls_answered AS DOUBLE)), 0) AS service_level_60s
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';
tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")
  ) |>
  summarise(
    service_level_60s =
      sum(pct_answered_within_60s * calls_answered, na.rm = TRUE) /
        sum(calls_answered, na.rm = TRUE)
  )

Interpretation notes. The result is a proportion between 0 and 1. Because only the weekly proportion is stored, the weighted average is an approximation of the true within-60s rate, but it is far closer than a simple average of weekly values.

Average Handle Time

The average total time agents spend per call, including any after-call wrap-up.

avg_handle_time_seconds covers more than avg_call_length_seconds: it adds wrap-up work after the caller disconnects. Like the other averages, combine weeks with a weighted mean.

Alternative names:

  • AHT
  • average handling time
  • total handle time per call
  • average talk plus wrap time

Calculation. Calculate a weighted average of avg_handle_time_seconds, weighted by calls_answered.

  • SQL
  • R
SELECT
    SUM(
        CAST(avg_handle_time_seconds AS DOUBLE)
        * CAST(calls_answered AS DOUBLE)
    )
    / NULLIF(SUM(CAST(calls_answered AS DOUBLE)), 0) AS weighted_avg_handle_time_seconds
FROM call_centre.weekly_activity_report
WHERE CAST(date_range_start AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-05-31';
tbl(con, in_schema("call_centre", "weekly_activity_report")) |>
  filter(
    as.Date(date_range_start) >= as.Date("2026-01-01"),
    as.Date(date_range_start) <= as.Date("2026-05-31")
  ) |>
  summarise(
    weighted_avg_handle_time_seconds =
      sum(avg_handle_time_seconds * calls_answered, na.rm = TRUE) /
        sum(calls_answered, na.rm = TRUE)
  )

Interpretation notes. Average handle time is always at least as large as average call length. A widening gap between the two points to growing after-call work.

Topic Metrics

Metrics in this section describe why people contact the call centre, sourced from call_centre.weekly_topic_dump. Because a single call can generate multiple topic rows, these metrics count topic records rather than distinct calls. Clean statobject before grouping, since it often carries leading, trailing, or repeated whitespace.

Call Topic Share

The percentage of all topic records represented by each call topic.

This is useful for understanding the relative importance of each call driver compared to the overall volume.

Alternative names:

  • share of calls by topic
  • percentage of calls by reason
  • topic distribution
  • call reason mix
  • contact driver share

Calculation. Count records by cleaned topic, then divide each topic count by the total count across all topics.

  • SQL
  • R
WITH topic_counts AS (
    SELECT
        REGEXP_REPLACE(TRIM(statobject), '\s+', ' ') AS call_topic,
        COUNT(*) AS topic_count
    FROM call_centre.weekly_topic_dump
    GROUP BY REGEXP_REPLACE(TRIM(statobject), '\s+', ' ')
)

SELECT
    call_topic,
    topic_count,
    CAST(topic_count AS DOUBLE)
        / NULLIF(SUM(topic_count) OVER (), 0) AS topic_share
FROM topic_counts
ORDER BY topic_count DESC;
tbl(con, in_schema("call_centre", "weekly_topic_dump")) |>
  mutate(
    call_topic = str_squish(statobject)
  ) |>
  count(call_topic, name = "topic_count") |>
  mutate(
    topic_share = as.double(topic_count) / sum(topic_count)
  ) |>
  arrange(desc(topic_count))

Interpretation notes. Topic share is useful for comparing relative demand across contact reasons. Interpret alongside total call volume.

Top 10 Call Topics

The ten most common call topics in the call centre topic data.

This is useful for quickly identifying the most common reasons people contact the call centre.

Alternative names:

  • top 10 call reasons
  • most common call topics
  • top contact drivers
  • main reasons for calling
  • top caller issues

Calculation. Count records by cleaned topic, calculate topic share, and return the top 10 by topic count.

  • SQL
  • R
WITH topic_counts AS (
    SELECT
        REGEXP_REPLACE(TRIM(statobject), '\s+', ' ') AS call_topic,
        COUNT(*) AS topic_count
    FROM call_centre.weekly_topic_dump
    GROUP BY REGEXP_REPLACE(TRIM(statobject), '\s+', ' ')
),

topic_shares AS (
    SELECT
        call_topic,
        topic_count,
        CAST(topic_count AS DOUBLE)
            / NULLIF(SUM(topic_count) OVER (), 0) AS topic_share
    FROM topic_counts
)

SELECT
    call_topic,
    topic_count,
    topic_share
FROM topic_shares
ORDER BY topic_count DESC
LIMIT 10;
tbl(con, in_schema("call_centre", "weekly_topic_dump")) |>
  mutate(
    call_topic = str_squish(statobject)
  ) |>
  count(call_topic, name = "topic_count") |>
  mutate(
    topic_share = as.double(topic_count) / sum(topic_count)
  ) |>
  slice_max(
    topic_count,
    n = 10,
    with_ties = FALSE
  ) |>
  arrange(desc(topic_count))

Interpretation notes. The top 10 view is useful for quick reporting but may hide important lower-volume issues. In cases where there are not many calls, it may be useful to limit this list to a top 5, or to truncate ties at the end of the list.
