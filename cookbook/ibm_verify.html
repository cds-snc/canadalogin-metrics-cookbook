IBM Verify

Metrics in this section are calculated from IBM Verify authentication and application sign-in data.

The relevant IBM Verify tables are:

  • ibm_verify.auth_total_logins - daily, CanadaLogin-wide sign-in totals
  • ibm_verify.app_login_counts - daily sign-in counts broken down by application
  • ibm_verify.mfa_activity - multi-factor authentication activity by factor and result

Internal applications. The following applications are typically excluded from public-facing adoption metrics:

'Flow Application',
'GCS Migration Solution',
'GC Sign In - Profile Management App'

CanadaLogin Metrics

Metrics in this section describe CanadaLogin overall, across all applications, sourced from ibm_verify.auth_total_logins.

Sign-Ins

The total number of sign-in events to CanadaLogin, including both successful and failed attempts.

Alternative names:

  • total sign-in attempts
  • sign-ins to CanadaLogin in a reporting period
  • total sign-ins to CanadaLogin

Calculation. Sum failed_logins and successful_logins over the selected reporting period.

  • SQL
  • R
SELECT
    SUM(failed_logins + successful_logins) AS total_sign_in_attempts
FROM ibm_verify.auth_total_logins
-- select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28';
tbl(con, in_schema("ibm_verify", "auth_total_logins")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  summarise(
    total_sign_in_attempts = sum(failed_logins + successful_logins, na.rm = TRUE)
  )

Sign-In Success Rate

The proportion of sign-in attempts that succeeded, across all of CanadaLogin.

This metric is useful for monitoring overall authentication health. A sustained drop can indicate a service problem, a misconfigured application, or elevated attack traffic inflating the failure count.

Alternative names:

  • authentication success rate
  • login success rate
  • overall sign-in success rate
  • successful sign-in proportion

Calculation. Divide the sum of successful_logins by the sum of all attempts (successful_logins + failed_logins) over the selected reporting period.

  • SQL
  • R
SELECT
    CAST(SUM(successful_logins) AS DOUBLE)
        / NULLIF(SUM(successful_logins + failed_logins), 0) AS sign_in_success_rate
FROM ibm_verify.auth_total_logins
-- select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28';
tbl(con, in_schema("ibm_verify", "auth_total_logins")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  summarise(
    sign_in_success_rate =
      sum(as.double(successful_logins), na.rm = TRUE) /
        sum(successful_logins + failed_logins, na.rm = TRUE)
  )

Interpretation notes. The rate is a proportion between 0 and 1. Read it alongside sign-in volume: a low success rate on very low volume may just be noise, while the same rate on high volume is a real signal.

Year-to-Date Unique Users

The total number of unique users who have signed in to CanadaLogin since the beginning of the current calendar year.

This metric is useful for understanding the overall reach and adoption of CanadaLogin over time. Note that this metric will not align with the sum of unique users across individual apps: a user who signs into App A and App B on the same day counts as two unique users at the app level but only one overall.

Alternative names:

  • total overall unique users this year
  • overall unique users YTD
  • year-to-date unique users of CanadaLogin

Calculation. Take the latest value of ytd_unique_users.

  • SQL
  • R
SELECT
    ytd_unique_users
FROM ibm_verify.auth_total_logins
ORDER BY from_date DESC
LIMIT 1;
tbl(con, in_schema("ibm_verify", "auth_total_logins")) |>
  slice_max(
    from_date,
    n = 1,
    with_ties = FALSE
  ) |>
  pull(ytd_unique_users)

Application Metrics

Metrics in this section describe sign-in activity broken down by application, sourced from ibm_verify.app_login_counts.

Sign-Ins by Application

The total number of sign-ins recorded for each application over a selected time period.

This metric is useful for understanding overall usage volume across CanadaLogin-integrated applications. It counts sign-in events, not people. A single user may sign in multiple times and contribute multiple sign-ins.

This metric is referred to in the source data as total_logins, but in reporting language it is usually clearer to describe it as total sign-ins.

Alternative names:

  • total sign-ins
  • sign-in volume
  • number of sign-ins
  • application sign-in count
  • total uses of CanadaLogin by application

Calculation. Sum total_logins over the selected reporting period, grouped by application.

  • SQL
  • R
SELECT
  application_name,
  SUM(total_logins) AS total_sign_ins
FROM   ibm_verify.app_login_counts
-- select an arbitrary reporting period
WHERE  from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
-- optionally filter out internal applications
  AND application_name NOT IN (
      'Flow Application',
      'GCS Migration Solution',
      'GC Sign In - Profile Management App'
  )
GROUP  BY application_name
ORDER  BY total_sign_ins DESC;
internal_apps <- c(
  "Flow Application",
  "GCS Migration Solution",
  "GC Sign In - Profile Management App"
)

tbl(con, in_schema("ibm_verify", "app_login_counts")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28"),
    # optionally filter out internal applications
    !application_name %in% internal_apps
  ) |>
  group_by(application_name) |>
  summarise(
    total_sign_ins = sum(total_logins, na.rm = TRUE)
  ) |>
  arrange(desc(total_sign_ins))

Interpretation notes. A sign-in count measures events, not people. Use this metric when the question is about system usage or activity volume. Use unique user metrics when the question is about reach.

Sign-In Success Rate by Application

The proportion of sign-in attempts that succeeded, broken down by application.

This metric helps isolate authentication problems to a specific application. An application with a markedly lower success rate than its peers is worth investigating for integration or configuration issues.

Alternative names:

  • authentication success rate by application
  • login success rate by application
  • per-application sign-in success rate

Calculation. Divide the sum of successful_logins by the sum of all attempts (successful_logins + failed_logins) over the selected reporting period, grouped by application.

  • SQL
  • R
SELECT
    application_name,
    CAST(SUM(successful_logins) AS DOUBLE)
        / NULLIF(SUM(successful_logins + failed_logins), 0) AS sign_in_success_rate
FROM ibm_verify.app_login_counts
-- select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
-- optionally filter out internal applications
  AND application_name NOT IN (
      'Flow Application',
      'GCS Migration Solution',
      'GC Sign In - Profile Management App'
  )
GROUP BY application_name
ORDER BY sign_in_success_rate DESC;
internal_apps <- c(
  "Flow Application",
  "GCS Migration Solution",
  "GC Sign In - Profile Management App"
)

tbl(con, in_schema("ibm_verify", "app_login_counts")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28"),
    # optionally filter out internal applications
    !application_name %in% internal_apps
  ) |>
  group_by(application_name) |>
  summarise(
    sign_in_success_rate =
      sum(as.double(successful_logins), na.rm = TRUE) /
        sum(successful_logins + failed_logins, na.rm = TRUE)
  ) |>
  arrange(desc(sign_in_success_rate))

Interpretation notes. Compare applications against each other rather than against an absolute target. Note that this measures authentication against a relying party after already signing-in to CanadaLogin. It doesn’t requiring entering a password or two-factor, it simply checks if a given credential is valid for signing in to an app.

Peak Daily Unique Users

The highest number of unique users observed on any single day during the reporting period.

This metric is useful for identifying the maximum daily reach of an application and for understanding peak demand.

Alternative names:

  • peak daily users
  • highest daily unique users
  • maximum daily active users
  • busiest day by unique users
  • peak daily reach

Calculation. Take the maximum value of unique_users over the selected reporting period, grouped by application.

  • SQL
  • R
WITH ranked AS (
    SELECT
        application_name,
        from_date,
        unique_users,
        ROW_NUMBER() OVER (
            PARTITION BY application_name
            ORDER BY unique_users DESC, from_date ASC
        ) AS rn
    FROM ibm_verify.app_login_counts
    -- calculate over an arbitrary reporting period
    WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
)

SELECT
    application_name,
    from_date,
    unique_users AS peak_daily_unique_users
FROM ranked
WHERE rn = 1
ORDER BY peak_daily_unique_users DESC;
tbl(con, in_schema("ibm_verify", "app_login_counts")) |>
  # calculate over an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  group_by(application_name) |>
  slice_max(
    unique_users,
    n = 1,
    with_ties = FALSE
  ) |>
  select(
    application_name,
    from_date,
    peak_daily_unique_users = unique_users
  ) |>
  arrange(desc(peak_daily_unique_users))

Interpretation notes. This metric identifies the busiest single day by unique user reach. The query above also returns the date on which that peak occurred.

Year-to-Date Unique Users

The number of unique users observed year-to-date for each application.

ytd_unique_users is a cumulative field that increases or remains stable over time. This metric is closer to “how many distinct people have used the application this year” than the daily unique user metrics.

Alternative names:

  • YTD unique users
  • unique users this year
  • year-to-date user reach
  • distinct users this year
  • total unique people reached this year

Calculation. Take the latest value of ytd_unique_users over the selected reporting period, grouped by application.

  • SQL
  • R
WITH ranked AS (
    SELECT
        application_name,
        from_date,
        ytd_unique_users,
        ROW_NUMBER() OVER (
            PARTITION BY application_name
            ORDER BY from_date DESC
        ) AS rn
    FROM ibm_verify.app_login_counts
    WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
)

SELECT
    application_name,
    ytd_unique_users
FROM ranked
WHERE rn = 1
ORDER BY ytd_unique_users DESC;
tbl(con, in_schema("ibm_verify", "app_login_counts")) |>
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  group_by(application_name) |>
  slice_max(
    from_date,
    n = 1,
    with_ties = FALSE
  ) |>
  select(
    application_name,
    ytd_unique_users
  ) |>
  arrange(desc(ytd_unique_users))

Interpretation notes. Since ytd_unique_users is a cumulative field, it will reset every year.

Average Daily Sign-Ins

The average number of sign-ins per day for a specific application over a selected reporting period.

This metric summarizes daily sign-in activity into a single average value. It is useful for understanding typical daily usage during a launch period, reporting window, or monitoring period.

Alternative names:

  • average sign-ins per day
  • typical daily sign-in volume
  • mean daily login volume
  • average daily CanadaLogin activity

Calculation. For each application, calculate the average of total_logins across all daily records in the selected reporting period.

  • SQL
  • R
SELECT
    application_name,
    AVG(total_logins) AS average_daily_sign_ins
FROM ibm_verify.app_login_counts
-- choose an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-04-22' AND DATE '2026-05-31'
GROUP BY application_name
ORDER BY average_daily_sign_ins DESC;
tbl(con, in_schema("ibm_verify", "app_login_counts")) |>
  # choose an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-04-22"),
    from_date <= as.Date("2026-05-31")
  ) |>
  group_by(application_name) |>
  summarise(
    average_daily_sign_ins = mean(total_logins, na.rm = TRUE)
  ) |>
  arrange(desc(average_daily_sign_ins))

Multi-factor Authentication Metrics

Metrics in this section describe multi-factor authentication (MFA) activity, sourced from ibm_verify.mfa_activity. That table has one row per day per MFA factor (mfa_type) per outcome (result), with the number of attempts in count.

MFA Authentications

The total number of MFA attempts over a selected reporting period.

This metric measures second-factor authentication volume. It counts attempts, not people, and includes both successful and failed challenges.

Alternative names:

  • MFA attempts
  • two-factor authentications
  • total MFA challenges
  • MFA volume

Calculation. Sum count over the selected reporting period. Add a result breakdown to separate successes from failures.

  • SQL
  • R
SELECT
    SUM(count) AS mfa_authentications
FROM ibm_verify.mfa_activity
-- select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28';
tbl(con, in_schema("ibm_verify", "mfa_activity")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  summarise(
    mfa_authentications = sum(count, na.rm = TRUE)
  )

MFA Success Rate

The proportion of MFA attempts that succeeded over a selected reporting period.

This metric is useful for monitoring how often users clear the second factor. A falling rate can point to user friction with a particular factor, delivery problems (for example, undelivered SMS or email OTPs), or attack traffic.

Alternative names:

  • MFA pass rate
  • second-factor success rate
  • MFA completion rate

Calculation. Divide the sum of count where result = 'success' by the sum of count across all results.

  • SQL
  • R
SELECT
    CAST(SUM(count) FILTER (WHERE result = 'success') AS DOUBLE)
        / NULLIF(SUM(count), 0) AS mfa_success_rate
FROM ibm_verify.mfa_activity
-- select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28';
tbl(con, in_schema("ibm_verify", "mfa_activity")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  summarise(
    mfa_success_rate =
      sum(if_else(result == "success", as.double(count), 0), na.rm = TRUE) /
        sum(count, na.rm = TRUE)
  )

Interpretation notes. The rate is a proportion between 0 and 1. Interpret per factor where possible: a low overall rate may be driven by a single struggling factor rather than a system-wide problem.

MFA Activity by Factor

The volume of MFA attempts broken down by factor type, with each factor’s share of total activity.

This metric shows which second factors users actually rely on. In the current data the mix is dominated by sms_otp and email_otp, with factors such as fido2, totp, kq, qr, and ibm_verify_push seeing little or no use.

Alternative names:

  • MFA factor mix
  • second-factor distribution
  • MFA usage by method
  • MFA factor share

Calculation. Sum count grouped by mfa_type, then divide each factor’s total by the overall total to get its share.

  • SQL
  • R
SELECT
    mfa_type,
    SUM(count) AS mfa_authentications,
    CAST(SUM(count) AS DOUBLE)
        / NULLIF(SUM(SUM(count)) OVER (), 0) AS factor_share
FROM ibm_verify.mfa_activity
-- select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
GROUP BY mfa_type
ORDER BY mfa_authentications DESC;
tbl(con, in_schema("ibm_verify", "mfa_activity")) |>
  # select an arbitrary reporting period
  filter(
    from_date >= as.Date("2026-01-01"),
    from_date <= as.Date("2026-05-28")
  ) |>
  group_by(mfa_type) |>
  summarise(
    mfa_authentications = sum(count, na.rm = TRUE)
  ) |>
  mutate(
    factor_share = as.double(mfa_authentications) / sum(mfa_authentications)
  ) |>
  arrange(desc(mfa_authentications))

Interpretation notes. factor_share is a proportion between 0 and 1 and sums to 1 across all factors. Combine with the MFA success rate to spot factors that are both heavily used and failing often.
