IBM Verify

Metrics in this section are calculated from IBM Verify authentication and application sign-in data.

Important Tables

  • ibm_verify.auth_total_logins contains info about CanadaLogin in general
  • ibm_verify.mfa_activity contains information about multi-factor authentication
  • ibm_verify.app_login_counts contains info about which apps users are authenticating against

Internal Applications

The following applications are typically excluded from public-facing adoption metrics:

'Flow Application',
'GCS Migration Solution',
'GC Sign In - Profile Management App'

Total Sign-Ins to CanadaLogin

Description

The total number of sign-in events to CanadaLogin. It includes both successful attempts and failed attempts to sign in.

Plain-language alternative names

  • Total sign-in attempts
  • Sign-ins to CanadaLogin in a reporting period
  • Total sign-ins to CanadaLogin

Data source

  • Schema: ibm_verify
  • Table: auth_total_logins

Calculation logic

The sum of failed_logins and successful_logins over a given reporting period.

Example SQL

SELECT 
    SUM(failed_logins + successful_logins) AS total_sign_in_attempts
FROM ibm_verify.auth_total_logins
-- Select an arbitrary reporting period
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
GROUP BY application_name
ORDER BY total_sign_in_attempts DESC;

Total Year-to-Date Unique Users of CanadaLogin

Description

The total number of unique users who have signed in to CanadaLogin since the beginning of the current calendar year.

This metric is useful for understanding the overall reach and adoption of CanadaLogin over time.

Note that this metric will not align with the sum of all of the unique users for each app. If I sign into App #1 and App #2 in the same day, I will count as a unique user in each app (which sums to 2) but only a single unique user overall.

Plain-language alternative names

  • Total overall unique users this year
  • Overall unique users YTD
  • Year-to-date unique users of CanadaLogin

Data source

  • Schema: ibm_verify
  • Table: auth_total_logins

Calculation logic

The latest value of the ytd_unique_users table.

Example SQL

SELECT
    ytd_unique_users
FROM ibm_verify.auth_total_logins
WHERE from_date >= DATE_TRUNC('year', CURRENT_DATE)
ORDER BY from_date DESC
LIMIT 1;

Total Sign-Ins by Application

Description

The total number of sign-ins recorded for each application over a selected time period.

This metric is useful for understanding overall usage volume across CanadaLogin-integrated applications. It counts sign-in events, not people. A single user may sign in multiple times and contribute multiple sign-ins.

This metric is currently referred to in the source data as total_logins, but in reporting language it is usually clearer to describe it as total sign-ins.

A sign-in count measures events, not people. Use this metric when the question is about system usage or activity volume.

Plain-language alternate names

  • Total sign-ins
  • Sign-in volume
  • Number of sign-ins
  • Application sign-in count
  • Total uses of CanadaLogin by application

Data source

  • Schema: ibm_verify
  • Table: app_login_counts

Calculation logic

Sum total_logins over the selected reporting period, grouped by application.

Example SQL

SELECT 
  application_name,
  SUM(total_logins) AS total_sign_ins
FROM   ibm_verify.app_login_counts
WHERE  from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
-- optionally filter out internal applications
  AND application_name NOT IN (
      'Flow Application',
      'GCS Migration Solution',
      'GC Sign In - Profile Management App'
  )
GROUP  BY application_name
ORDER  BY total_sign_ins DESC;

Peak Daily Users by Application

Description

The highest number of unique users observed on any single day during the reporting period.

This metric is useful for identifying the maximum daily reach of an application and for understanding peak demand.

Unlike total daily unique users, this metric does not sum across days. It identifies the single highest daily value.

This metric is useful for identifying the busiest single day by unique user reach.

Plain-language alternate names

  • Peak daily users
  • Highest daily unique users
  • Maximum daily active users
  • Busiest day by unique users
  • Peak daily reach

Data source

  • Schema: ibm_verify
  • Table: app_login_counts

Calculation logic

Take the maximum value of unique_users over the selected reporting period, grouped by application.

Example SQL

SELECT
    application_name,
    MAX(unique_users) AS peak_daily_unique_users
FROM ibm_verify.app_login_counts
WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
GROUP BY application_name
ORDER BY peak_daily_unique_users DESC;

Year-to-Date Unique Users by Application

Description

The number of unique users observed year-to-date for each application.

Generally, this is calculated by taking the last value of ytd_unique_users over the reporting period or within a year. ytd_unique_users is a cumulative field that increases or remains stable over time.

This metric is closer to “how many distinct people have used the application this year” than the daily unique user metrics.

Since ytd_unique_users is a cumulative field, it will reset every year.

Plain-language alternate names

  • YTD unique users
  • Unique users this year
  • Year-to-date user reach
  • Distinct users this year
  • Total unique people reached this year

Data source

  • Schema: ibm_verify
  • Table: app_login_counts

Calculation logic

Take the maximum value of ytd_unique_users over the selected reporting period, grouped by application.

Example SQL

WITH ranked AS (
    SELECT
        application_name,
        from_date,
        ytd_unique_users,
        ROW_NUMBER() OVER (
            PARTITION BY application_name
            ORDER BY from_date DESC
        ) AS rn
    FROM ibm_verify.app_login_counts
    WHERE from_date BETWEEN DATE '2026-01-01' AND DATE '2026-05-28'
)

SELECT
    application_name,
    ytd_unique_users AS ytd_unique_users
FROM ranked
WHERE rn = 1
ORDER BY ytd_unique_users DESC;

Average Daily Sign-Ins by Application

Description

The average number of sign-ins per day for a specific application over a selected reporting period.

This metric summarizes daily sign-in activity into a single average value. It is useful for understanding typical daily usage during a launch period, reporting window, or monitoring period.

For example, this metric can be used to describe the average daily sign-in volume for PrairiesCan between April 22, 2026 and May 31, 2026.

Plain-language alternate names

  • Average daily sign-ins
  • Average sign-ins per day
  • Typical daily sign-in volume
  • Mean daily login volume
  • Average daily CanadaLogin activity

Data source

  • Schema: ibm_verify
  • Table: app_login_counts

Calculation logic

For a given reporting period, calculate total sign-ins by day, then take the average of those daily totals.

Example SQL

WITH daily_sign_ins AS (
    SELECT
        application_name,
        from_date,
        total_logins
    FROM ibm_verify.app_login_counts
    -- choose an arbitrary reporting period
    WHERE from_date BETWEEN DATE '2026-04-22' AND DATE '2026-05-31'
    GROUP BY
        application_name,
        from_date
)

SELECT
    application_name,
    AVG(total_logins) AS average_daily_sign_ins
FROM daily_sign_ins
GROUP BY application_name
ORDER BY average_daily_sign_ins DESC;
