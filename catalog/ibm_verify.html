IBM Verify

Derived tables in the ibm_verify schema, populated from the IBM Verify reporting API. Each table is a daily aggregate derived from the raw event stream in ibm_verify_raw.ibm_verify. The field-level meaning of those raw events is documented separately in the IBM Verify event data dictionary, organized by event type (Authentication Event, MFA Authentication Event, SSO, and so on).

Data retention

The dataset behind the IBM Verify reporting API retains only the most recent 366.5 days of data. This does not limit the tables here. The daily aggregate rows are stored on our side and kept beyond a year, so any metric we calculate by aggregating those stored daily rows (sign-ins, MFA attempts, and other additive measures) is unaffected, no matter how far back the window reaches.

The limit bites only on summaries that the API itself computes, because the API can only see the last 366.5 days when it does so. The cumulative unique-user columns are the affected case: a count must be computed by the API rather than summed from daily rows (summing daily unique users would count user-days, not distinct people). The mtd_ and ytd_ windows fit inside the retention horizon, but a “since the April 22, 2026 launch” total silently truncates once launch is more than a year in the past (roughly late April 2027). The since_apr22_unique_users fields that previously carried it have therefore been removed from the pipeline.

app_login_counts

Daily sign-in counts for each CanadaLogin-integrated application.

Each row represents one application on one day, summarizing that application’s sign-in activity and unique-user reach for the day.

Keywords: sign-ins, logins, applications, unique users, adoption, daily counts

Provenance

Queried directly from the IBM Verify production instance via API.

  • Updated: Daily, at 6am America/Toronto
  • Steward: EDCP Data and Research Team
  • Contact: Yinlin Zhang

Fields

One row per application_name per day.

Field Name Type Description
application_name string Name of the integrated application (for example, “IRIS - PRC - Prod”)
from_date timestamp Start of the reporting day (midnight)
to_date timestamp End of the reporting day (23:59:59)
total_logins integer Total sign-in events recorded for the application that day. Counts events, not people. Reported as “total sign-ins”
unique_users integer Number of distinct users who signed in to the application that day
failed_logins integer Number of failed sign-in attempts that day
successful_logins integer Number of successful sign-ins that day
mtd_unique_users integer Cumulative distinct users for the application month-to-date. Resets at the start of each month
ytd_unique_users integer Cumulative distinct users for the application year-to-date. Resets at the start of each calendar year
rolling_365d_unique_users integer Distinct users who signed in to this application in the trailing 365 days ending on that row’s date
fetched_at timestamp When the aggregation last fetched this row
date string The reporting day as a text date (for example, “2026-04-23”). Mirrors from_date

Notes

total_logins, unique_users, failed_logins, and successful_logins are daily values. mtd_unique_users, ytd_unique_users, and rolling_365d_unique_users are cumulative counts of distinct users and should not be summed across rows; take the latest value within the window instead.

rolling_365d_unique_users is null before 2026-06-13, its implementation date.

A since_apr22_unique_users field (cumulative distinct users since the April 22, 2026 public launch) was previously provided here. It has been removed from the pipeline because the source API retains only the most recent 366.5 days of data, so a since-launch cumulative count cannot be maintained once launch is more than a year in the past. See the retention note on the IBM Verify catalog overview.

Unlike auth_total_logins, these counts are not inflated by second factors. The per-authentication-method event inflation described there only happens when a user signs in to CanadaLogin itself. Authenticating against an integrated application reuses the credentials already established in the CanadaLogin session, so the user enters no password or second factor and a single application sign-in records a single event. total_logins and successful_logins still count events rather than people, because one person can sign in to an application repeatedly. Use unique_users when the question is about how many distinct people used the application.

auth_total_logins

Daily, CanadaLogin-wide sign-in totals across all applications.

Each row represents one day and summarizes overall CanadaLogin authentication activity. Unlike app_login_counts, there is no per-application breakdown.

Keywords: sign-ins, logins, totals, CanadaLogin overall, unique users, daily

Provenance

Queried directly from the IBM Verify production instance via API.

  • Updated: Daily, at 6am America/Toronto
  • Steward: EDCP Data and Research Team
  • Contact: Yinlin Zhang

Fields

One row per day.

Field Name Type Description
from_date timestamp Start of the reporting day (midnight)
to_date timestamp End of the reporting day (23:59:59)
unique_users integer Number of distinct users who signed in to CanadaLogin that day. In IBM Verify, this is referred to as “unique logins”.
failed_logins integer Number of failed sign-in attempts that day
successful_logins integer Number of successful authentication events that day.
mtd_unique_users integer Cumulative distinct users month-to-date. Resets at the start of each month
ytd_unique_users integer Cumulative distinct users year-to-date. Resets at the start of each calendar year
rolling_365d_unique_users integer Distinct users who signed in to CanadaLogin in the trailing 365 days ending on that row’s date
last_updated_at timestamp When the underlying data for this row was last updated
fetched_at timestamp When the aggregation last fetched this row
date string The reporting day as a text date (for example, “2026-05-21”). Mirrors from_date

Notes

unique_users, failed_logins, and successful_logins are daily values. The mtd_, ytd_, and rolling_365d_ unique-user fields are cumulative and should not be summed across rows; take the latest value within the window instead.

rolling_365d_unique_users is null before 2026-06-13, its implementation date.

A since_apr22_unique_users field (cumulative distinct users since the April 22, 2026 launch) was previously provided here. It has been removed from the pipeline because the source API retains only the most recent 366.5 days of data, so a since-launch cumulative count cannot be maintained once launch is more than a year in the past. See the retention note on the IBM Verify catalog overview.

When a user signs in to CanadaLogin itself, IBM Verify records each authentication method as a separate event. Entering a username and password followed by a second factor like an OTP counts as two successful events. A mistyped password that is then entered correctly, followed by a correct OTP, counts as three events: two successful and one failed. Both successful_logins and failed_logins are therefore inflated relative to the number of distinct sign-in sessions.

This inflation applies only to signing in to CanadaLogin itself. Authenticating against an integrated application does not inflate these counts, because the user reuses the credentials already established in their CanadaLogin session and enters no password or second factor (see app_login_counts).

Summing successful_logins is therefore a misleading indicator of actual user engagement. Instead, use the unique_users field, which is daily active users, and is a measure of the number of users who engaged with the system in a given day.

mfa_activity

Daily multi-factor authentication (MFA) activity, broken down by factor type and outcome.

Each row represents the count of MFA attempts for a single combination of day, MFA factor type, and result.

Keywords: MFA, multi-factor, second factor, OTP, FIDO2, authentication, security

Provenance

Queried directly from the IBM Verify production instance via API.

  • Updated: Daily, at 6am America/Toronto
  • Steward: EDCP Data and Research Team
  • Contact: Yinlin Zhang

Fields

One row per day per mfa_type per result.

Field Name Type Description
from_date timestamp Start of the reporting day (midnight)
to_date timestamp End of the reporting day (23:59:59)
mfa_type string The MFA factor used. One of “sms_otp”, “fido2”, “kq”, “totp”, “voice_otp”, “qr”, “ibm_verify_push”, “email_otp”
result string Outcome of the MFA event. One of “success”, “failure”, or “sent”
count integer Number of MFA attempts for this day, factor, and result
fetched_at timestamp When the aggregation last fetched this row
date string The reporting day as a text date (for example, “2026-05-31”). Mirrors from_date

Notes

Only OTP factor types (sms_otp, email_otp, voice_otp) generate a “sent” event, because they are issued before the user responds. Successfully clearing one of these challenges therefore counts as two events: first a “sent” event, then either a “success” or “failure” event depending on how the user responds. It is possible to generate multiple “success”/“failure” events on a single “sent” event if the user inputs their OTP incorrectly first and then correctly. Non-OTP factor types (fido2, kq, totp, ibm_verify_push) have no “sent” event and produce only “success”/“failure” rows.

When calculating MFA usage metrics, de-duplicate accordingly: count “sent” events for OTP factors (sending an email OTP but not using it is still usage), and “success”/“failure” events for non-OTP factors. Summing “sent” alone would silently drop every non-OTP factor.

When calculating success rate metrics, use the number of successful events divided by the sum of the success and failure events, excluding “sent”.
