Platform

Platform metrics for CanadaLogin, in the sre schema. Currently, both tables are derived from the CanadaLogin Platform team’s CanadaLogin_MTTR_Dashboard Google Sheet (maintained by hand in the CDS Google Drive) and fetched by the sre-mttr-sheets-fetcher Lambda using Google Sheets API 6am ET every Monday. They share one source and grain (one row per incident); they differ only in history: mttr_current holds the latest state of each incident, while mttr_history accumulates a dated snapshot of every incident on each upload.

Data quality

Because the source is a hand-maintained spreadsheet, a few caveats apply to both tables:

  • Timestamps are strings, not native timestamps. detected_at, acknowledged_at, mitigated_at, and resolved_at are text in America/Toronto local time at minute precision (YYYY-MM-DD HH:MM). Parse them before doing date math. (fetched_at is a real timestamp, in UTC.)
  • URLs may carry trailing whitespace or embedded newlines. Trim slack_channel and postmortem_link before joining on or rendering them.
  • Categorical columns are unvalidated free text. severity, service, component, and root_cause_category are typed by hand, so expect occasional typos, casing, and spacing drift. The values listed in each field table are those observed so far, not a closed set.
  • The duration columns are rounded to the nearest minute from second-precision source times, so recomputing them from the displayed (minute-precision) timestamp strings can differ by up to a minute.

mttr_current

The latest known state of every CanadaLogin incident, one row per incident.

Each row is a single incident. The table is overwritten on each upload with the most recent values, so it never grows beyond the current incident set. For point-in-time history, see mttr_history.

Keywords: MTTR, MTTA, MTTM, incident, incident response, postmortem, severity, reliability, SRE, mean time to resolve

Provenance

Fetched automatically every Monday at 06:00 ET from the CanadaLogin MTTR Dashboard Google Sheet using the sre-mttr-sheets-fetcher Lambda. Each upload overwrites this table wholesale with the current incident state and appends a dated snapshot to mttr_history.

  • Updated: Weekly, Every Monday 06:00 ET
  • Steward: EDCP Data and Research Team
  • Contact: Yinlin Zhang (Data and Research team) or Obie Okongwu (CL Platform Team)

Fields

One row per incident, keyed by incident_id. See the schema-level Data quality note for caveats that apply to the string timestamps, URL columns, free-text categories, and rounded durations.

Field Name Type Description
incident_id string Unique identifier for the incident, format INC-NNN
slack_channel string URL of the incident’s dedicated Slack channel
severity string Incident severity. One of “Sev1”, “Sev2”, “Sev3”, or “Sev4” (Sev1 most severe, Sev4 least). Free text in the source, not strictly enforced
service string The CanadaLogin service affected
component string The specific subsystem affected, more granular than service. Free text
short_description string One-line free-text summary of the incident
detected_at string When the incident was detected, America/Toronto local time at minute precision (YYYY-MM-DD HH:MM)
acknowledged_at string When the incident was acknowledged. Same format and timezone as detected_at
mitigated_at string When the incident was mitigated. Same format and timezone as detected_at
resolved_at string When the incident was fully resolved. Same format and timezone as detected_at
root_cause_category string Free-text root-cause classification
mtta_alert_ack integer Minutes from detected_at to acknowledged_at, rounded to the nearest minute
mttm_detected_mitigated integer Minutes from detected_at to mitigated_at, rounded to the nearest minute
mttr_detected_resolved integer Minutes from detected_at to resolved_at, rounded to the nearest minute
postmortem_link string URL of the incident’s postmortem Google Doc. Optional
week integer ISO-8601 week number (1-53) of detected_at
month string Calendar month of detected_at, first-of-month formatted as mmm-yy text (for example “Feb-26”)
fetched_at timestamp When the pipeline ingested this snapshot, in UTC

mttr_history

The append-only companion to mttr_current: same source, columns, and provenance, but instead of holding only the latest state it accumulates a dated snapshot of every incident on each upload.

Each row is one incident (incident_id) as it appeared on one snapshot date, so an incident’s metrics can change from one snapshot to the next as it is re-triaged. A row is unique on (incident_id, date).

Keywords: MTTR, MTTA, MTTM, incident, incident response, snapshot, history, severity, reliability, SRE, mean time to resolve

Fields

Every column is identical to mttr_current, with one addition:

Field Name Type Description
date string Snapshot date as YYYY-MM-DD text; the grain that lets this table accumulate. Equals the date portion of fetched_at

Uptime Tables

The uptime tables are sourced from the cds-snc/status-statut public GitHub repository and updated automatically each day. No manual steps required.

uptime_summary

One row per monitored site per day. Each row is a snapshot of rolling-window statistics fetched from history/summary.json in the cds-snc/status-statut GitHub repository at 04:00 ET.

Provenance

Fully automated. No manual upload required. The upptime summary.yml GitHub Action recalculates all rolling-window metrics from the full five-minute ping history once per day (~00:00 UTC), rewriting summary.json. The Lambda runs at 04:00 ET and fetches that file, writing one Parquet row per site into the date=YYYY-MM-DD partition corresponding to the preceding day.

  • Updated: Daily, at ~04:00 ET
  • Steward: EDCP Data and Research Team
  • Contact: Yinlin Zhang (Data and Research team)

Fields

Field Name Type Description
date varchar Hive partition key: the date the Lambda ran for (yesterday’s date, YYYY-MM-DD).
fetch_date varchar UTC timestamp when the Lambda fetched the data (YYYY-MM-DD HH:MM:SS). Useful for diagnosing staleness.
slug varchar Upptime site identifier. One of the six values in the table above.
lang varchar Short label identifying the site variant, one of the six values in the table above.
url varchar The monitored URL.
status varchar Site status at fetch time: up or down.
uptime_day double Uptime percentage over the past 24 hours (~288 five-minute pings).
uptime_week double Uptime percentage over the past 7 days (~2,016 pings).
uptime_month double Uptime percentage over the past 30 days (~8,640 pings).
uptime_year double Uptime percentage over the past 365 days (~105,120 pings).
uptime_alltime double Uptime percentage across all pings since monitoring began (April 2, 2026).
response_day_ms bigint Average response time over the past 24 hours, in milliseconds.
response_week_ms bigint Average response time over the past 7 days, in milliseconds.
response_month_ms bigint Average response time over the past 30 days, in milliseconds.
response_year_ms bigint Average response time over the past 365 days, in milliseconds.
response_alltime_ms bigint Average response time across all pings since monitoring began, in milliseconds.
daily_minutes_down varchar JSON string mapping each outage date to total minutes down on that date.

Notes

Timing note. When the Lambda runs on June 30, it fetches rolling-window statistics last recalculated at ~23:2x UTC on June 29. The resulting row lands in partition date=2026-06-29. All rolling windows trail backward from that reference point.

uptime_timeseries

One row per upptime bot commit per site. The upptime bot commits to the cds-snc/status-statut history YAML files once per day (~23:2x UTC) as a daily snapshot, plus additional commits immediately when a status change is detected mid-day.

Provenance

Fully automated. No manual upload required. Sourced from the GitHub Commits API for each site’s history file (history/<slug>.yml).

  • Updated: Daily, at ~04:00 ET
  • Steward: EDCP Data and Research Team
  • Contact: Yinlin Zhang (Data and Research team)

Expected row count per site per day:

Scenario Expected rows
Normal all-up day 1 (daily ~23:2x UTC snapshot, status = 'up')
Outage day 2+ (down event + recovery event + daily snapshot)

Fields

Field Name Type Description
date varchar Hive partition key: the UTC date of the commit (YYYY-MM-DD). Use in WHERE date = '...' or WHERE CAST(date AS DATE) BETWEEN ... filters.
ping_timestamp varchar UTC timestamp of the upptime bot’s commit (YYYY-MM-DD HH:MM:SS). On a normal day this is the daily snapshot at ~23:2x UTC. On an outage day earlier commits record when the site went down or recovered.
slug varchar Upptime site identifier, same six values as in sre.uptime_summary.
lang varchar Short label, same six values as in sre.uptime_summary.
status varchar Ping result at the moment of this commit: up, down, or unknown.
http_code bigint HTTP status code at the moment of this commit. 200 on a normal day; 0 or 503 when down. null if unparseable.
response_ms bigint Response time in milliseconds at the moment of this commit. Single-ping measurement at commit time, not a daily average. null if unparseable.

Notes

response_ms is a single-ping measurement. It records the response time at the exact moment of the upptime bot’s commit, not a daily average. For averaged response times use sre.uptime_summary (response_day_ms, response_week_ms, etc.), which are computed from all 288 five-minute pings.

status = 'down' rows count detection events, not outage duration. Each down row is one commit, one moment when upptime detected a failure. For total minutes of downtime on a given date, use the daily_minutes_down JSON column in sre.uptime_summary.
