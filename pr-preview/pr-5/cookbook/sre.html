Platform

Site reliability metrics for the CanadaLogin platform, in the sre schema. Currently incident-response metrics, with other reliability measures to follow.

The relevant tables are:

  • sre.mttr_current - latest state of every incident, one row per incident
  • sre.mttr_history - dated snapshots of every incident, preserving how each evolved

These tables come from a hand-maintained spreadsheet; see the catalog Data quality note for caveats. The one that shapes most queries here: the incident timestamps (detected_at, resolved_at, and so on) are stored as America/Toronto strings, not native dates, so a reporting window filters on the YYYY-MM-DD prefix of detected_at rather than a from_date column.

Incident Response Metrics

Mean Time to Resolution

The average time, in minutes, from when an incident is detected to when it is fully resolved.

Alternative names:

  • MTTR
  • time to resolve
  • incident resolution time
  • mean time to recovery (see the interpretation note on this overloaded term)

Calculation. Average mttr_detected_resolved (minutes from detected_at to resolved_at, precomputed in the table) across the incidents whose detected_at falls in the reporting window, from sre.mttr_current. Because the incident timestamps are stored as America/Toronto strings rather than native dates, the window casts the YYYY-MM-DD prefix of detected_at to a DATE instead of filtering a from_date column. Unresolved incidents have a null mttr_detected_resolved and are excluded from the average.

To break the figure out for service-level reporting, group the same average by severity.

  • SQL
  • R

Overall:

SELECT
    AVG(mttr_detected_resolved) AS mean_time_to_resolution
FROM sre.mttr_current
-- set the reporting window on detected_at; see the timespans note in cookbook/notes.qmd
WHERE CAST(substr(detected_at, 1, 10) AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-06-26';

By severity:

SELECT
    severity,
    AVG(mttr_detected_resolved) AS mean_time_to_resolution
FROM sre.mttr_current
-- set the reporting window on detected_at; see the timespans note in cookbook/notes.qmd
WHERE CAST(substr(detected_at, 1, 10) AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-06-26'
GROUP BY severity
ORDER BY severity;

Overall:

tbl(con, in_schema("sre", "mttr_current")) |>
  # set the reporting window on detected_at; see the timespans note in cookbook/notes.qmd
  filter(
    as.Date(str_sub(detected_at, 1, 10)) >= as.Date("2026-01-01"),
    as.Date(str_sub(detected_at, 1, 10)) <= as.Date("2026-06-26")
  ) |>
  summarise(
    mean_time_to_resolution = mean(mttr_detected_resolved, na.rm = TRUE)
  )

By severity:

tbl(con, in_schema("sre", "mttr_current")) |>
  # set the reporting window on detected_at; see the timespans note in cookbook/notes.qmd
  filter(
    as.Date(str_sub(detected_at, 1, 10)) >= as.Date("2026-01-01"),
    as.Date(str_sub(detected_at, 1, 10)) <= as.Date("2026-06-26")
  ) |>
  group_by(severity) |>
  summarise(
    mean_time_to_resolution = mean(mttr_detected_resolved, na.rm = TRUE)
  ) |>
  arrange(severity)

Interpretation notes. The figure is in minutes; divide by 60 for hours. It is a mean, so a single very long incident can pull it up sharply, especially given how few incidents there are in any window. Read it alongside the incident count, and consider a median or a Sev1-only cut when one outlier dominates.

The window is anchored on detected_at, so an incident counts in the period it began, even if it was resolved in a later period. severity is unvalidated free text in the source sheet, so the grouped view reflects whatever values were typed (expect the occasional typo or spacing drift).

Site Uptime Metrics

Uptime and response-time metrics for the six monitored CanadaLogin sites, in the sre schema. Data comes from the cds-snc/status-statut GitHub repository and is fetched automatically by the uptime-daily-download Lambda at 04:00 ET each day. No manual steps are required.

The relevant tables are:

  • sre.uptime_summary - one row per site per day; rolling-window uptime percentages and average response times
  • sre.uptime_timeseries - one row per upptime bot commit per site; individual ping events including outage detections

Both tables are partitioned by date=YYYY-MM-DD. Because date is a varchar (derived from the Hive partition path), cast it before doing date arithmetic: CAST(date AS DATE) in SQL; as.Date(date) in R.

Site Uptime

The rolling-window uptime percentage for each monitored CanadaLogin site, showing what proportion of five-minute pings succeeded over each time horizon.

Six sites are monitored: the two public-facing login pages (EN/FR), the Manage Profile app (staging and production), and the IdP (staging and production).

Alternative names:

  • uptime percentage
  • availability percentage
  • site availability
  • service uptime by site

Calculation. Filter to the most recent date partition and read the precomputed rolling-window uptime columns from sre.uptime_summary. Each uptime_* column is already a percentage (0-100), calculated from all five-minute pings over the named window.

To track uptime over time for a specific site, remove the MAX(date) filter and group by date.

  • SQL
  • R

Latest snapshot, all sites, all rolling windows:

SELECT
    slug,
    lang,
    status,
    uptime_day,
    uptime_week,
    uptime_month,
    uptime_year,
    uptime_alltime
FROM sre.uptime_summary
WHERE date = (SELECT MAX(date) FROM sre.uptime_summary)
ORDER BY slug;

Trend over time, year uptime for a specific site:

SELECT
    date,
    uptime_year,
    uptime_alltime
FROM sre.uptime_summary
-- set the reporting window; see the timespans note in cookbook/notes.qmd
WHERE slug = 'canada-login-website-en'
  AND CAST(date AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-06-30'
ORDER BY date;

Latest snapshot, all sites, all rolling windows:

uptime_s <- tbl(con, in_schema("sre", "uptime_summary"))
latest_date <- uptime_s |> summarise(d = max(date, na.rm = TRUE)) |> pull(d)

uptime_s |>
  filter(as.Date(date) == as.Date(latest_date)) |>
  select(slug, lang, status, uptime_day, uptime_week, uptime_month,
         uptime_year, uptime_alltime) |>
  arrange(slug)

Trend over time, year uptime for a specific site:

tbl(con, in_schema("sre", "uptime_summary")) |>
  # set the reporting window; see the timespans note in cookbook/notes.qmd
  filter(
    slug == "canada-login-website-en",
    as.Date(date) >= as.Date("2026-01-01"),
    as.Date(date) <= as.Date("2026-06-30")
  ) |>
  select(date, uptime_year, uptime_alltime) |>
  arrange(date)

Interpretation notes. The uptime windows are rolling, not calendar-aligned. uptime_month on a given date covers the 30 days trailing that date’s stats update (~23:2x UTC), not a fixed calendar month. To aggregate by calendar month, group by substr(date, 1, 7).

Site Response Time

The average HTTP response time for each monitored CanadaLogin site, aggregated over rolling windows from five-minute ping data.

Alternative names:

  • average response time
  • site latency
  • response latency by site
  • HTTP response time

Calculation. Filter to the most recent date partition and read the precomputed rolling-window response time columns from sre.uptime_summary. Each response_*_ms column is the mean response time in milliseconds over all five-minute pings in the named window.

For a single-ping point-in-time reading (not an average), use sre.uptime_timeseries.response_ms instead.

  • SQL
  • R

Latest snapshot, all sites, all rolling windows:

SELECT
    slug,
    lang,
    status,
    response_day_ms,
    response_week_ms,
    response_month_ms,
    response_year_ms,
    response_alltime_ms
FROM sre.uptime_summary
WHERE date = (SELECT MAX(date) FROM sre.uptime_summary)
ORDER BY slug;

Trend over time, weekly average response time for a specific site:

SELECT
    date,
    response_week_ms,
    response_alltime_ms
FROM sre.uptime_summary
-- set the reporting window; see the timespans note in cookbook/notes.qmd
WHERE slug = 'canada-login-website-en'
  AND CAST(date AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-06-30'
ORDER BY date;

Latest snapshot, all sites, all rolling windows:

uptime_s <- tbl(con, in_schema("sre", "uptime_summary"))
latest_date <- uptime_s |> summarise(d = max(date, na.rm = TRUE)) |> pull(d)

uptime_s |>
  filter(as.Date(date) == as.Date(latest_date)) |>
  select(slug, lang, status, response_day_ms, response_week_ms,
         response_month_ms, response_year_ms, response_alltime_ms) |>
  arrange(slug)

Trend over time, weekly average response time for a specific site:

tbl(con, in_schema("sre", "uptime_summary")) |>
  # set the reporting window; see the timespans note in cookbook/notes.qmd
  filter(
    slug == "canada-login-website-en",
    as.Date(date) >= as.Date("2026-01-01"),
    as.Date(date) <= as.Date("2026-06-30")
  ) |>
  select(date, response_week_ms, response_alltime_ms) |>
  arrange(date)

Interpretation notes. Response times are averages computed from all 288 five-minute pings in the window. Single-measurement spikes are smoothed out. During outage periods, pings that time out contribute high response times or are excluded depending on the upptime implementation, so treat response time trends during known outage windows with caution.

Outage Events

Individual site outage events detected by the uptime monitor: when a site went down, when it recovered, and the HTTP status code at time of detection.

Alternative names:

  • downtime events
  • site outages
  • availability incidents
  • uptime incidents

Calculation. Filter sre.uptime_timeseries to rows where status = 'down'. Each such row represents a moment when the upptime bot detected the site was unreachable and committed that state to the history file.

To count outage events per site over a reporting period, group the filtered rows by slug and count.

  • SQL
  • R

All outage events in a reporting period:

SELECT
    date,
    ping_timestamp,
    slug,
    lang,
    http_code,
    response_ms
FROM sre.uptime_timeseries
-- set the reporting window; see the timespans note in cookbook/notes.qmd
WHERE status = 'down'
  AND CAST(date AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-06-30'
ORDER BY date DESC, slug;

Outage event count per site in a reporting period:

SELECT
    slug,
    lang,
    COUNT(*) AS outage_event_count
FROM sre.uptime_timeseries
-- set the reporting window; see the timespans note in cookbook/notes.qmd
WHERE status = 'down'
  AND CAST(date AS DATE) BETWEEN DATE '2026-01-01' AND DATE '2026-06-30'
GROUP BY slug, lang
ORDER BY outage_event_count DESC;

All outage events in a reporting period:

tbl(con, in_schema("sre", "uptime_timeseries")) |>
  # set the reporting window; see the timespans note in cookbook/notes.qmd
  filter(
    status == "down",
    as.Date(date) >= as.Date("2026-01-01"),
    as.Date(date) <= as.Date("2026-06-30")
  ) |>
  select(date, ping_timestamp, slug, lang, http_code, response_ms) |>
  arrange(desc(date), slug)

Outage event count per site in a reporting period:

tbl(con, in_schema("sre", "uptime_timeseries")) |>
  # set the reporting window; see the timespans note in cookbook/notes.qmd
  filter(
    status == "down",
    as.Date(date) >= as.Date("2026-01-01"),
    as.Date(date) <= as.Date("2026-06-30")
  ) |>
  count(slug, lang, name = "outage_event_count") |>
  arrange(desc(outage_event_count))

Interpretation notes. Each row in sre.uptime_timeseries is a single commit by the upptime bot. On a normal all-up day there is one row per site (the daily ~23:2x UTC snapshot). On an outage day there are additional rows (one when the site went down and one when it recovered), so status = 'down' rows count detection events, not total minutes of downtime. For total outage duration, use the daily_minutes_down JSON field in sre.uptime_summary.
